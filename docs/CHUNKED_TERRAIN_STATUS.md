# Chunked Terrain LOD System - Implementation Status

## ✅ Completed Components

### 1. Core Architecture
- **ChunkManager**: Resource that manages all terrain chunks
- **ChunkId**: Unique identifier for each chunk (face_index, lod_level, chunk_x, chunk_y)
- **TerrainChunk**: Component storing chunk state and mesh handles
- **ChunkGenerationTask**: Async task for generating chunk meshes

### 2. LOD System
- **Distance-based LOD calculation**: Calculates appropriate LOD level based on camera distance
- **Variable resolution**: Chunks at different LOD levels use different mesh resolutions
  - LOD 0 (closest): 64x64 vertices per chunk
  - Higher LODs: Progressively lower resolution (32x32, 16x16, etc.)
- **Logarithmic LOD distribution**: Smooth transitions between detail levels

### 3. Chunk Generation
- **Async mesh generation**: Uses `AsyncComputeTaskPool` for non-blocking generation
- **Cube sphere chunking**: Divides each of the 6 cube faces into chunks
- **Terrain noise integration**: Uses existing `TerrainNoise` system for consistent terrain
- **Collider generation**: Creates mesh colliders for low-LOD chunks (performance optimization)

### 4. Chunk Management
- **Dynamic loading**: Chunks are generated on-demand based on camera position
- **Unloading system**: Distant chunks are automatically unloaded to save memory
- **Frame budget**: Limits chunk generation to 3 chunks per frame to maintain performance

## 🔧 Current Limitations & Next Steps

### Issue 1: Integration with Existing Planet System
**Problem**: The chunked terrain system and the existing `PlanetPlugin` both try to manage planet rendering. Currently, both systems run simultaneously, which may cause conflicts.

**Solution Options**:
1. **Option A**: Add a feature flag to disable old planet generation when chunked terrain is enabled
2. **Option B**: Make chunked terrain replace the old planet mesh after initial generation
3. **Option C**: Keep both systems but hide the old planet mesh when chunks are loaded

**Recommended**: Option A - Add a `use_chunked_terrain: bool` flag to `PlanetSettings` and conditionally disable old generation.

### Issue 2: Detail Texture Sharing
**Problem**: Chunks need access to the detail texture generated by `PlanetPlugin`, but currently they use a default empty texture.

**Current Status**: `ChunkManager` has a `detail_texture_handle` field, but it's not being populated from the planet generation.

**Solution**: Modify `check_planet_generation` in `planet.rs` to store the texture handle in `ChunkManager` when planet generation completes.

### Issue 3: Chunk Boundary Seams
**Problem**: Chunks at different LOD levels may have visible seams where they meet.

**Solution**: Implement geomorphing (smooth transitions) between LOD levels:
- Adjust vertices at chunk boundaries to match neighboring chunks
- Use vertex shader to blend between LOD levels
- This is marked as TODO in the roadmap

### Issue 4: Performance Tuning
**Current Settings**:
- `max_chunks_per_frame: 3` - May be too conservative
- `max_active_chunks: 200` - May need adjustment based on testing
- `base_chunk_resolution: 64` - May need to be higher for better detail

**Next Steps**: Profile and tune these values based on actual performance.

## 🎯 How It Works

### Chunk Generation Flow:
1. **Camera Position Check**: Every frame, `update_chunk_lods` checks camera position
2. **LOD Calculation**: Determines which LOD level is needed for each cube face
3. **Chunk Spawning**: Spawns chunk entities for needed chunks (if not already loaded)
4. **Async Generation**: Starts async task to generate mesh and collider
5. **Task Processing**: `process_chunk_generation_tasks` completes up to 3 tasks per frame
6. **Mesh Attachment**: When generation completes, mesh and material are attached to chunk entity

### LOD Calculation:
```rust
// Distance-based LOD
let lod_level = calculate_lod_level(
    distance_to_face,
    min_lod_distance,  // radius + 100m
    max_lod_distance,  // radius * 3
    max_lod_levels     // 6 levels
);
```

### Chunk Resolution:
- LOD 0: 64x64 vertices (highest detail)
- LOD 1: 32x32 vertices
- LOD 2: 16x16 vertices
- LOD 3+: 8x8 vertices (minimum)

## 📊 Performance Characteristics

### Memory Usage:
- **Per Chunk**: ~1-5MB (depending on LOD level)
- **Max Active**: 200 chunks = ~200MB-1GB (worst case)
- **Typical**: ~50-100 chunks = ~50-500MB

### Generation Time:
- **Per Chunk**: ~1-10ms (CPU-bound, async)
- **Per Frame**: Up to 3 chunks = ~3-30ms (spread across frames)
- **Initial Load**: ~100-200 chunks = ~100ms-2s (non-blocking)

### Rendering:
- **Triangles**: Varies by LOD (64² = 4K triangles at LOD 0)
- **Draw Calls**: One per chunk (can be optimized with instancing)
